# 序列与非序列统一建模相关方案地图

## 1. 总体问题定义

TAAC2026 的核心不是单纯做序列推荐，也不是单纯做 CTR 特征交叉，而是：

> 在一个统一架构里，同时处理多字段行为序列、用户/物品非序列特征、上下文特征，并输出 pCVR。

这类问题可以理解为三个技术流派的交汇：

1. **CTR 特征交互模型**：DeepFM、xDeepFM、DCN、AutoInt、RankMixer、Wukong
2. **行为序列建模模型**：DIN、DIEN、DSIN、BST、SIM、ETA、SDIM、LongCTR
3. **统一架构模型**：HyFormer、OneTrans、Hiformer/HiFormer、生成式推荐框架

---

## 2. 方案一：传统 Late Fusion / 拼接融合

### 思路

```text
序列特征 -> Pooling / RNN / Transformer -> user_seq_emb
非序列特征 -> Embedding + MLP -> dense_emb
目标物品特征 -> Embedding -> item_emb

concat(user_seq_emb, dense_emb, item_emb) -> MLP -> pCVR
```

### 代表模型

- Wide & Deep
- DeepFM
- PNN
- xDeepFM
- DCN / DCN-V2

### 优点

- 工程简单
- 稳定、易复现
- 对小数据和中等复杂度任务很实用

### 缺点

- 序列信息被提前压缩成一个向量
- 非序列特征无法细粒度影响序列建模过程
- 目标物品与历史行为的交互发生太晚

### 对 TAAC2026 的启发

这类方法可以作为轻量 baseline 或 ensemble 子模型，但不是赛题鼓励的主要方向。官方 baseline 已经明显超越了 Late Fusion。

---

## 3. 方案二：Target Attention 目标感知序列建模

### 思路

用候选物品作为 query，去关注用户历史行为中与目标物品最相关的部分。

```text
target_item_emb = Embedding(target item)
history_embs = Embedding(user behavior sequence)

attention_weight = Attention(target_item_emb, history_embs)
interest_emb = weighted_sum(history_embs)

concat(interest_emb, target_item_emb, other_features) -> MLP -> pCTR/pCVR
```

### 代表模型

| 模型 | 重点 |
|---|---|
| DIN | Deep Interest Network，目标感知注意力 |
| DIEN | Deep Interest Evolution Network，引入兴趣演化 GRU |
| DSIN | Deep Session Interest Network，把行为切成 session |
| BST | Behavior Sequence Transformer，用 Transformer 建模行为序列 |

### 优点

- 目标物品可以直接选择相关历史行为
- 比简单 pooling 更强
- 工业 CTR 场景成熟可靠

### 缺点

- 多数模型仍然是“序列模块 + 特征模块”的组合
- 非序列特征与序列特征交互仍偏晚
- 多行为域、多字段序列场景下结构会变复杂

### 对 TAAC2026 的启发

官方 `HyFormer` 中的 sequence query 与 non-sequence token 交互，可以看成 Target Attention 的升级版：不只是 target item attend history，而是所有序列 query 和非序列 token 做多轮深度交互。

## 4. 方案三：Session / Multi-Interest 建模

### 思路

用户兴趣通常不是单一向量，而是多个兴趣簇或多个 session。

```text
历史行为序列 -> 切分 session / 动态路由 / 多头注意力
              -> interest_1, interest_2, ..., interest_k

target item 与多个 interest 匹配 -> 得分
```

### 代表模型

| 模型 | 重点 |
|---|---|
| MIND | Capsule Network + 动态路由，提取多兴趣 |
| ComiRec | KDD 2020，多兴趣召回，含 ComiRec-SA / ComiRec-DR |
| DSIN | session-aware，分会话建模 |
| DMIN | Deep Match to Interest Network，多兴趣匹配 |

### 优点

- 适合用户兴趣多样化
- 对长历史行为比单向量 pooling 更合理
- 可解释性更强：不同 interest token 对应不同兴趣方向

### 缺点

- 兴趣数量选择敏感
- 训练复杂度更高
- 如果只有 pCVR 排序目标，召回式多兴趣结构不一定直接最优

### 对 TAAC2026 的启发

`num_queries` 本质上就是给每个序列域提取多个兴趣 token。可以尝试：

- `num_queries=1/2/4` 做消融
- 不同 domain 使用不同 query 数量（需要改代码）
- 用 session 统计特征辅助序列建模

## 5. 方案四：长序列建模

### 问题

真实推荐系统里用户历史行为可能非常长，标准 Transformer 的复杂度是 `O(L^2)`，不可直接扩展。

### 代表方法

| 方法 | 机制 |
|---|---|
| SIM | Search-based Interest Model，从长历史中检索相关行为 |
| ETA | Efficient Target Attention，高效目标注意力 |
| SDIM | Sampling Is All You Need，采样长期行为 |
| TWIN | 多阶段长序列建模 |
| LongCTR | 长序列 CTR benchmark，收集多种长序列方案 |
| LONGER | 长序列高效编码，OneTrans/HyFormer 文献中常被提及 |

### 常见技术路线

1. **截断**：只保留最近 N 条，简单但可能丢失长期兴趣
2. **检索**：根据 target item 检索最相关历史行为
3. **采样**：按时间、类别、重要性采样
4. **压缩**：用聚类、池化或 Top-K 注意力压缩
5. **分层**：先 session 内编码，再 session 间编码

### 对 TAAC2026 的启发

官方 baseline 中已有：

- `seq_max_lens`: 控制 A/B/C/D 序列长度
- `seq_encoder_type=longer`: 长序列编码器
- `seq_top_k`: LongerEncoder 保留 Top-K

后续调参方向：

```bash
--seq_encoder_type longer \
--seq_top_k 50 \
--seq_max_lens seq_a:256,seq_b:256,seq_c:512,seq_d:512
```

## 6. 方案五：特征交互 Scaling

### 背景

近年来工业推荐模型发现：简单扩大 DLRM/MLP 参数不一定有效，真正有用的是扩大“特征交互模块”的容量。

### 代表模型

| 模型 | 重点 |
|---|---|
| AutoInt | 自注意力学习显式特征交互 |
| DCN-V2 | CrossNet 显式高阶交叉 |
| xDeepFM | CIN 显式向量级交互 |
| Wukong | 面向 Scaling Law 的模态推荐/特征交互 |
| RankMixer | 工业排序模型中的 token mixing 交互模块 |

### RankMixer 思路

```text
features -> tokenized features
         -> token mixing across feature tokens
         -> per-token FFN
         -> ranking prediction
```

### 对 TAAC2026 的启发

官方 baseline 的 `rank_mixer_mode=full` 是非常关键的交互模块。可以做：

- `full` vs `ffn_only` vs `none` 消融
- 增大 `d_model`
- 增大 token 数量：`user_ns_tokens`, `item_ns_tokens`, `num_queries`
- 但注意显存和训练速度

## 7. 方案六：统一 Transformer 架构

### 代表论文

| 论文/模型 | 核心思想 |
|---|---|
| HyFormer | 重新审视序列建模与特征交互在 CTR 中的角色，用 HyFormer Block 统一建模 |
| OneTrans | 用一个 Transformer 同时做 feature interaction 和 sequence modeling |
| Hiformer / HiFormer | 异构特征交互，用 Transformer 处理 heterogeneous features |

### OneTrans 的重要观点

OneTrans 指出：

- 过去 RankMixer/Wukong 主要扩大 feature interaction 模块
- LONGER 等主要扩大 sequence modeling 模块
- 两条线分开扩展，会阻碍双向信息交换
- 应该用一个统一 Transformer stack 同时建模

### HyFormer 与 OneTrans 的差别理解

| 维度 | HyFormer | OneTrans |
|---|---|---|
| 核心机制 | Sequence Query + NS Token 交互 | 所有特征与序列在一个 Transformer 中统一建模 |
| 工程倾向 | 更注重可控复杂度 | 更彻底统一，但可能更重 |
| 与官方 baseline | 高度一致 | 可作为后续改进方向 |

### 对 TAAC2026 的启发

官方 baseline `PCVRHyFormer` 已经站在统一建模方向上，后续创新可以围绕：

1. 更好的 NS Tokenizer
2. 更好的 sequence query 生成方式
3. 更高效的 cross-attention
4. 更强的 RankMixer/Transformer block
5. 多任务或生成式目标

## 8. 方案七：生成式推荐

### 背景

2025 腾讯广告算法大赛强调 All-Modality Generative Recommendation。生成式推荐把 item / content 映射到离散 token 空间，然后用自回归模型预测下一个交互。

### 代表思路

```text
item_id / 文本 / 图片 / 协同行为 -> 离散 token
user history token sequence -> autoregressive LM
生成下一个 item token / ad token
```

### 优点

- 端到端生成推荐结果
- 可以融合文本、图像、多模态内容
- 有机会突破传统排序范式

### 缺点

- 与当前 pCVR baseline 差异大
- 训练和评估工程更复杂
- 初期不建议作为第一优先级

### 对 TAAC2026 的启发

当前第一阶段先跑通官方 pCVR baseline。生成式推荐可以作为中长期探索方向，尤其当赛题后续引入更多模态或序列生成要求时。

## 10. 当前阶段推荐优先级

### P0：先跑通官方 baseline

不要急着改模型。先确认：

- train 任务能跑起来
- ckpt 能保存
- model 能 publish
- eval 能生成 predictions.json
- leaderboard 有分数

### P1：做最小代价消融

基于 baseline，只改 `run.sh` 参数：

1. `num_queries`: 1 / 2 / 4
2. `user_ns_tokens`: 4 / 5 / 8
3. `item_ns_tokens`: 2 / 4
4. `seq_encoder_type`: transformer / longer
5. `rank_mixer_mode`: full / ffn_only
6. `loss_type`: bce / focal

### P2：数据理解与特征统计

等平台跑通后，利用训练日志和可访问的数据，统计：

- label 正负比例
- 各 domain 序列长度分布
- 各离散特征 vocab size
- 缺失值比例
- user/item 热度分布

### P3：结构改进

如果 baseline 稳定后再考虑代码级修改：

1. 改 NS Tokenizer
2. 改 sequence query 生成方式
3. 引入 target-aware attention
4. 加 session-level pooling
5. 加高阶交叉/门控融合
