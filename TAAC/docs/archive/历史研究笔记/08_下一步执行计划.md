## Phase 3：第一轮最小消融实验

只改 `run.sh` 参数，不改代码。

### 实验 A：默认 baseline

```bash
--ns_tokenizer_type rankmixer
--user_ns_tokens 5
--item_ns_tokens 2
--num_queries 2
--rank_mixer_mode full
--seq_encoder_type transformer
```

目标：拿到第一个有效分数。

### 实验 B：更轻量版本

```bash
--d_model 48
--batch_size 256
--num_hyformer_blocks 2
```

目标：更快验证，降低 OOM 风险。

### 实验 C：增加序列表达

```bash
--num_queries 4
```

观察：

- 分数是否提升
- 显存是否明显增加
- 训练速度是否变慢

### 实验 D：增加非序列表达

```bash
--user_ns_tokens 8
--item_ns_tokens 4
```

观察：

- 非序列信息保留更多是否有收益
- RankMixer token 数增加后是否过拟合

### 实验 E：长序列编码器

```bash
--seq_encoder_type longer
--seq_top_k 50
```

观察：

- 如果官方数据序列很长，可能有收益
- 如果序列不长，收益可能有限

### 实验 F：Focal Loss

```bash
--loss_type focal
--focal_alpha 0.1
--focal_gamma 2.0
```

适合正负样本极不平衡的 pCVR 场景。

### 今晚两卡实验（2026-04-29）

卡 1：适度放宽高基数序列 embedding。

```bash
--seq_encoder_type swiglu
--emb_skip_threshold 5000000
--ns_tokenizer_type rankmixer
--user_ns_tokens 5
--item_ns_tokens 2
--num_queries 2
--d_model 64
--num_heads 4
--batch_size 128
--seq_max_lens "seq_a:128,seq_b:128,seq_c:256,seq_d:256"
--num_epochs 8
--patience 4
--sparse_lr 0.05
--dropout_rate 0.01
--reinit_sparse_after_epoch 0
--reinit_cardinality_threshold 0
--amp
--amp_dtype bfloat16
```

已知失败配置：

```bash
--emb_skip_threshold 0
```

该配置会创建全部 embedding，日志显示约 `10,341,768,961` 总参数、`10,339,307,392` sparse 参数，并在 `torch.optim.Adagrad` 初始化 state tensor 时 OOM。不要再直接使用 0。若 `5000000` 仍在 optimizer 初始化 OOM，退到 `2000000`。

卡 2：使用 `ns_groups.json` 语义 NS token。

```bash
--seq_encoder_type swiglu
--ns_tokenizer_type group
--ns_groups_json "${SCRIPT_DIR}/ns_groups.json"
--num_queries 1
--d_model 64
--num_heads 4
--emb_skip_threshold 1000000
--batch_size 128
--seq_max_lens "seq_a:128,seq_b:128,seq_c:256,seq_d:256"
--num_epochs 8
--patience 4
--sparse_lr 0.05
--dropout_rate 0.01
--reinit_sparse_after_epoch 0
--reinit_cardinality_threshold 0
--amp
--amp_dtype bfloat16
```

eval 注意事项：

- 不需要为这两组实验分别修改 `eval/infer.py`，前提是 checkpoint 目录内存在 `train_config.json`。
- 卡 2 还必须确认 checkpoint 目录内有 `ns_groups.json`，否则 eval 端会退化为 singleton groups 或 fallback，导致结构不一致。
- `_FALLBACK_MODEL_CFG` 只能兜底默认配置，不能同时匹配两个实验；不要依赖 fallback 跑官方 eval。

### 2026-04-30 结果解读与下一轮方向

关键信号：

- `emb_skip_threshold=5000000` 将官方 eval 提到 `0.809921`，是当前最好结果。
- `group tokenizer + num_queries=1` 的 valid AUC 最高，但官方 eval 只有 `0.805905`，再次证明 valid/test 关系弱。
- group 实验不能简单判定为“语义分组无效”，因为它同时把每个序列域 query 从 2 降到 1，降低了序列表达容量。

下一轮优先方向：

1. 固定 `swiglu + rankmixer + num_queries=2 + full reinit`，继续搜索 `emb_skip_threshold`：`10000000`，若 optimizer 初始化 OOM 则退到 `7000000`。
2. 在当前最佳 `emb_skip_threshold=5000000` 上恢复更长序列：`seq_a:256,seq_b:256,seq_c:512,seq_d:512`。
3. group tokenizer 若要继续，应使用 `d_model=80,num_queries=2,num_heads=4`，保持序列 query 容量；但这不是下一轮第一优先级。

## Phase 5：如果 baseline 无法跑通，优先排查

### 1. 找不到 schema.json

说明没有正确选择官方数据集，或数据路径不是官方训练数据目录。

### 2. import 失败

检查上传包是否漏文件：

- `dataset.py`
- `model.py`
- `utils.py`
- `trainer.py`

### 3. eval 导入失败

检查 eval 包中文件名是否正确：

```text
dataset.py
model.py
```

不要是：

```text
dataset (1).py
model (1).py
```

### 4. OOM

优先改：

```bash
--batch_size 128
--seq_max_lens seq_a:128,seq_b:128,seq_c:256,seq_d:256
--amp
--amp_dtype bfloat16
```

如果仍在第一步 forward OOM，再降到：

```bash
--batch_size 64
--seq_max_lens seq_a:64,seq_b:64,seq_c:128,seq_d:128
```

最后再考虑 `d_model=48` 或减少 HyFormer block，因为这会改变模型结构和可比性。

### 5. predictions.json 报错

检查：

- user_id 是否转为字符串
- 是否包含所有测试用户
- 是否多出无关 user_id
- 概率是否在 0~1
