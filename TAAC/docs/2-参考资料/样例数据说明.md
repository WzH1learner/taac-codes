# 样例数据说明

本文是 `sample_data/README_data.md` 的中文精炼版，用于说明 TAAC2026 demo 数据的字段结构和建模含义。

## 1. 数据概览

| 项目 | 内容 |
|---|---|
| 文件 | `sample_data/demo_1000.parquet` |
| 样本数 | 1,000 |
| 列数 | 120 |
| 格式 | Parquet，扁平列布局 |

数据使用 flat column layout，所有特征都是顶层列，不再使用嵌套结构。

## 2. 字段类别

| 类别 | 数量 | 数据类型 | 说明 |
|---|---:|---|---|
| ID & Label | 5 | `int64` / `int32` | 核心 ID、标签和时间戳 |
| User Int Features | 46 | `int64` / `list<int64>` | 用户离散特征，包含标量和数组 |
| User Dense Features | 10 | `list<float>` | 用户连续向量特征 |
| Item Int Features | 14 | `int64` / `list<int64>` | 物品离散特征 |
| Domain Sequence Features | 45 | `list<int64>` | 4 个行为域的序列特征 |

## 3. ID 与标签列

| 列名 | 类型 | 说明 |
|---|---|---|
| `user_id` | `int64` | 用户 ID |
| `item_id` | `int64` | 物品 ID |
| `label_type` | `int32` | 标签类型 |
| `label_time` | `int64` | 标签时间 |
| `timestamp` | `int64` | 样本曝光/事件时间 |

当前训练代码中：

```text
label = (label_type == 2)
```

即 `label_type == 2` 为正样本，其余为负样本。

## 4. 用户特征

User Int：

- 标量列：`user_int_feats_{1,3,4,48-59,82,86,92-109}`，共 35 列。
- 数组列：`user_int_feats_{15,60,62-66,80,89-91}`，共 11 列。

User Dense：

- `user_dense_feats_{61-66,87,89-91}`，共 10 列。

官方说明中提到：当 `user_int_feats_{fid}` 与 `user_dense_feats_{fid}` 拥有相同 fid 时，它们通常共同描述同一实体或信号。

## 5. 物品特征

Item Int：

- 标量列：`item_int_feats_{5-10,12-13,16,81,83-85}`，共 13 列。
- 数组列：`item_int_feats_{11}`，共 1 列。

当前 dataset 侧 `item_dense_feats` 为空，因此 item 侧有效输入主要来自这些离散特征。

## 6. 序列特征

四个行为域：

| 行为域 | 列名范围 | 列数 |
|---|---|---:|
| `seq_a` | `domain_a_seq_{38-46}` | 9 |
| `seq_b` | `domain_b_seq_{67-79,88}` | 14 |
| `seq_c` | `domain_c_seq_{27-37,47}` | 12 |
| `seq_d` | `domain_d_seq_{17-26}` | 10 |

代码会根据 `schema.json` 中每个 domain 的 `ts_fid` 识别时间戳列，并生成 `{domain}_time_bucket`。

## 7. 对优化工作的启发

- 序列长度普遍远超当前截断长度，但 `seq*2` official eval 未提升，说明不能只增加长度。
- 更值得做的是 recency、interval、recent-window count 和 item-history match。
- 修改序列输入字段时，必须同步 `train/dataset.py` 与 `eval/dataset.py`。
