# 推荐算法优化蓝图

最后更新：2026-05-05

本文是 TAAC2026 PCVRHyFormer 后续优化的唯一可信源。它整合当前实验结论、平台约束、代码现实和外部高分经验，用于指导后续开发、评审、训练和 official eval 决策。

## 1. 背景与目标

### 1.1 比赛问题

TAAC2026 统一推荐挑战赛当前任务是 pCVR 预估：输入用户画像、物品画像和多域行为序列，输出点击后转化概率。官方评分以 eval AUC 为准，训练日志中的 validation AUC 只能作为辅助信号。

### 1.2 平台约束

- 训练入口是 `train/run.sh`，平台通过环境变量传入 `TRAIN_DATA_PATH`、`TRAIN_CKPT_PATH`、`TRAIN_LOG_PATH`。
- 推理入口必须是 `eval/infer.py`，必须生成 `predictions.json`。
- 每个连续 24 小时最多 3 个评估任务，失败或停止任务不计入次数；额度重置时间按平台说明为游戏时间 15:59:59。
- 结构性模型改动必须同时同步 `train/` 和 `eval/`，否则 eval 可能出现 shape mismatch 或缺字段。

### 1.3 当前目标

短期目标不是追求更高 validation AUC，而是稳定超过当前 official eval 最优：

```text
official eval AUC = 0.809921
```

当前可回退主基线：

```text
seq_encoder_type = swiglu
ns_tokenizer_type = rankmixer
emb_skip_threshold = 5000000
seq_max_lens = seq_a:128,seq_b:128,seq_c:256,seq_d:256
loss_type = bce
sparse_lr = 0.05
dropout_rate = 0.01
reinit_sparse_after_epoch = 0
reinit_cardinality_threshold = 0
valid_split_mode = tail
checkpoint_select_metric = auc
```

## 2. 代码现实

### 2.1 数据结构

sample 数据包含 1000 行、120 列：

| 类别 | 数量 | 说明 |
|---|---:|---|
| ID & Label | 5 | `user_id`, `item_id`, `label_type`, `label_time`, `timestamp` |
| User Int | 46 | 标量或数组离散特征 |
| User Dense | 10 | 数组连续特征 |
| Item Int | 14 | 标量或数组离散特征 |
| Domain Sequence | 45 | 4 个行为域序列特征 |

训练代码中 `label_type == 2` 被视为正样本。`item_dense_feats` 当前为空张量，item 侧主要依赖 `item_int_feats`。

sample 序列长度远超当前截断长度：

| 序列域 | sample mean | p50 | p90 | max | 当前截断 |
|---|---:|---:|---:|---:|---:|
| seq_a | 701 | 577 | 1562 | 1888 | 128 |
| seq_b | 571 | 405 | 1393 | 1952 | 128 |
| seq_c | 449 | 322 | 887 | 3894 | 256 |
| seq_d | 1100 | 1035 | 2215 | 3951 | 256 |

但是 `seq*2` 虽然提升 validation AUC，却几乎不提升 official eval，说明问题不是简单“序列太短”，而是序列信息如何被模型使用。

### 2.2 模型结构

核心路径：

```text
user/item int + user dense -> NS tokens
seq_a/b/c/d -> seq tokens + time bucket embedding
NS tokens + seq mean -> MultiSeqQueryGenerator -> query tokens
query tokens + NS tokens -> MultiSeqHyFormerBlock -> RankMixer -> classifier
```

关键现实：

- `MultiSeqQueryGenerator` 使用 `NS_flat + seq_mean` 生成 query，缺少 recency、interval、target item matching 等显式信息。
- `SwiGLUEncoder` 是当前最稳的序列编码器；相同 5M 阈值下 Transformer official eval 更差。
- `RankMixerNSTokenizer` 把所有 embedding 拼接后均分成固定 token，user 侧可能切到 feature embedding 内部，item 侧当前刚好较整齐。
- `rank_mixer_mode=full` 要求 `d_model % T == 0`，其中 `T = num_queries * num_sequences + num_ns`。
- `LongerEncoder` 代码注释目标是取 latest top-k，但当前实现按 `valid_len - top_k` 取尾部；sample timestamp 看起来更像 newest-first，因此该模块在正式实验前必须先确认序列方向。

### 2.3 训练与保存逻辑

- 稀疏 Embedding 使用 `Adagrad`，稠密参数使用 `AdamW(betas=(0.9,0.98))`。
- 当前没有 lr scheduler / warmup。
- `checkpoint_select_metric=auc` 时按 validation AUC 保存 best model；`last` 已被实验否定。
- `reinit_sparse_after_epoch=0` 且 `reinit_cardinality_threshold=0` 不是关闭 reinit，而是每个 epoch 后重置所有正基数 Embedding。该策略当前仍是主线的一部分。

### 2.4 Train/Eval 同步规则

需要同步 `train/` 与 `eval/` 的改动：

| 改动类型 | 需要同步 eval |
|---|---|
| 新增 `ModelInput` 字段 | 是 |
| 新增 dataset 输出张量 | 是 |
| 新增 model constructor 参数 | 是 |
| 新增结构性超参 | 是，且要写入 `train_config.json` / `infer.py` fallback |
| lr、warmup、loss、valid split、checkpoint 策略 | 否 |

## 3. 已完成实验结论

### 3.1 有效或仍值得跟进

| 实验 | 关键配置 | official eval | 判断 |
|---|---|---:|---|
| 当前主基线 | SwiGLU + RankMixer + 5M + BCE + short seq | 0.809921 | 当前最优 |
| SwiGLU full reinit | 相比 Transformer 只换 SwiGLU | 0.806956 | SwiGLU 优于 Transformer old baseline |
| emb_skip_threshold=5M | 相比 1M 恢复部分高基数字段 | 0.809921 | 有效，固定为主线 |
| sparse_lr=0.08 | 仅 `0.05 -> 0.08` | 待 official eval | 训练健康，优先提交 |

`sparse_lr=0.08` 的关键训练信号：

```text
best valid AUC = 0.861314@epoch6
valid LogLoss = 0.224980
valid prob_mean = 0.096718
valid label_mean = 0.096785
```

### 3.2 已否定或暂停

| 路线 | 结果 | 结论 |
|---|---:|---|
| `time_window=2h` | 0.7883 | 大幅下降，不作为训练/选模协议 |
| `train_include_valid=True + last` | 0.80387 | validation 泄漏膨胀，不提升 eval |
| `use_seq_time_delta_proj=True` | 0.809214 | 单个 per-token delta 投影不够 |
| `Transformer + 5M` | 0.80656 | 不回退 Transformer |
| `emb_skip_threshold=7M` | 0.806861 | 参数翻倍但 eval 下降 |
| `seq*2` | 0.809907 | valid 高但 eval 不涨，且推理慢 |
| `focal_alpha=0.25` | 未提交 | AUC 尚可但 prob_mean 过高，校准差 |

### 3.3 重要方法论结论

1. validation AUC 与 official eval 的相关性有限，valid 只能作为训练健康度和候选筛选信号。
2. 单纯增大模型容量、序列长度或阈值，已经进入低收益区。
3. 高分经验反复指向：optimizer/warmup、item 侧、pair 特征、序列行为时间戳处理。
4. 时间特征不是“裸加 hour/week bucket”。外部经验显示直接加小时/星期可能掉点；更合理的是相对 recency、interval、recent-window intensity。

## 4. 已规划优化点

### P0. 优化器与 Warmup

依据：

- 外部高分经验明确提到 `scale + lr warmup` / optimizer。
- 当前代码没有 scheduler。
- `sparse_lr=0.08` 训练曲线和校准都健康。

可执行方案：

1. 先提交 `sparse_lr=0.08` 的 official eval。
2. 增加 dense lr warmup：前 `0.5-1 epoch` 从小 lr 线性升到 `1e-4`。
3. 如果 P0 有收益，再尝试 sparse lr warmup 或 `sparse_lr=0.10`。

改动范围：

- `train/train.py`：新增 `--warmup_epochs` 或 `--warmup_steps`。
- `train/trainer.py`：每 step 更新 optimizer lr。
- 不改 `eval/`。

验证标准：

- validation AUC 不崩。
- LogLoss 与 prob_mean/label_mean 不明显变差。
- official eval 超过 `0.809921` 才继续扩展该方向。

### P1. 序列 Recency / Interval 统计特征

依据：

- 测试集时间窗极窄，eval 诊断显示样本集中在约 1.55 小时窗口。
- 当前已有 time bucket embedding，但 query 生成只看 `seq_mean`。
- `seq*2` 不涨说明“更长”不等于“更会用最近行为”。

可执行方案：

1. 每个 domain 输出统计特征：`len_norm`、最近行为距今、recent-window count / ratio。
2. 统计窗口建议先用 `5m/30m/2h/1d/7d`。
3. 增加 interval 特征：相邻行为时间间隔的 `min/mean/max/log_mean`。
4. 把这些 stats 作为 query generator 的补充输入，或投影成额外 domain stats token。

改动范围：

- `train/dataset.py` 与 `eval/dataset.py`：生成 `seq_time_stats`。
- `train/model.py` 与 `eval/model.py`：新增小投影层，并接入 query/global context。
- `eval/infer.py`：补 fallback config。

验证标准：

- 不要求 valid AUC 大涨，但校准不能明显变差。
- official eval 和推理耗时是最终判断。

### P2. Item 侧与 Pair 特征

依据：

- PCVR 是 user-item interaction 任务。
- 当前 item 侧只通过 item NS token 进入，缺少显式 target-history match。
- 外部高分经验明确提到 item 侧与 pair 特征。

可执行方案：

1. 对 target item int 与各序列域 side-info 做 exact match 统计：是否出现、出现次数、最近一次出现距今。
2. 做 recent-window pair count：`2h/1d/7d` 内 match 次数。
3. 加 item/user 特征质量统计：zero count、有效字段数、疑似 cold-start 指示。

改动范围：

- `train/dataset.py` 与 `eval/dataset.py`：生成 `pair_dense_feats`。
- `train/model.py` 与 `eval/model.py`：`pair_dense_proj` 作为额外 token 或拼到 classifier 前。
- 需要先用 sample/schema 识别 item fids 与 seq fids 的候选对应关系。

验证标准：

- 先单独消融，不与 P1/P0 混跑。
- 如果 valid AUC 一般但校准健康，仍可 official eval 一次，因为 pair 特征常更贴 test。

### P3. 修正并验证 LongerEncoder

依据：

- sample 中部分序列 timestamp 头部看起来更新，可能是 newest-first。
- `LongerEncoder` 当前按尾部取 top-k，可能取到更旧行为。
- 它理论上比 Transformer 更适合长序列，但必须先修正“最近 top-k”定义。

可执行方案：

1. 用平台 schema/sample 确认四个 domain 的顺序方向。
2. 增加 `--seq_order newest_first|oldest_first` 或直接修正 `_gather_top_k`。
3. 运行 `seq_encoder_type=longer --seq_top_k 64` 的干净对照。

优先级：

低于 P1/P2。只有在序列 stats 或 pair 特征推进后，再投入 official eval。

### P4. 配置治理

依据：

`train/run.sh` 当前 active config 与最佳基线存在漂移风险：

```text
emb_skip_threshold=1000000
ns_groups_json=""
num_epochs=20
patience=15
```

这些值不应作为下一轮默认冲榜配置。

可执行方案：

1. 把 active block 改成当前主基线或最新确认最佳。
2. 每个实验块写清楚：改动目的、变量、是否需要同步 eval。
3. 每次实验记录 official eval、valid AUC、LogLoss、prob_mean/label_mean、推理耗时。

## 5. 潜在盲区与未验证假设

### 5.1 我们可能高估了 validation 的作用

多次实验显示 valid AUC 高不一定 eval 高。`seq*2` 和 `train_include_valid` 都强化了这个结论。后续不要用单一 valid AUC 做选模依据，应至少加入：

- LogLoss / Brier。
- prob_mean 与 label_mean 距离。
- 推理耗时。
- 是否与已知 test 诊断一致。

### 5.2 时间切分没有真正解决 test mismatch

`time_tail` 与 `time_window` 都没有带来收益。原因可能不是切分方式简单错误，而是官方 test 是特殊小时/星期/曝光窗口切片。继续换 holdout 的收益低，应该转向 test-aware 特征工程和无标签分布诊断。

### 5.3 语义分组没有被公平证伪

旧 group 实验混入 `num_queries=1`；后续语义混合又混入 `d_model=80`、`time_tail`、`focal`。因此不能下结论“semantic group 一定不行”，只能说它不是当前最高优先级。

如要重试，必须使用：

```text
group + BCE + tail-valid + threshold=5M + d_model=80 + num_queries=2
```

### 5.4 Focal Loss 不是无效，而是当前 alpha 可能不对

`focal_alpha=0.25/0.5` 导致预测均值明显高于正样本率。后续若重试，应从更低 alpha 或 gamma 开始，并把校准作为 official eval 前置条件。

### 5.5 `emb_skip_threshold` 的真实问题是“全有或全无”

当前超过阈值的超高基数字段被置零，低于阈值则建完整 embedding。`5M` 有收益、`7M` 变差说明某些高基数字段有用，但完整恢复会带来噪声和参数爆炸。这里可能需要更细粒度的高基数字段处理，而不是继续粗调阈值。

### 5.6 `run.sh` 漂移会污染实验

平台默认执行 `run.sh`。如果 active config 不是最佳基线，队友很容易在错误基线之上继续实验。每次提交前必须确认 `run.sh`、平台参数、`train_config.json` 三者一致。

## 6. 新优化方向提议

### N1. Hash Embedding for Skipped High-Cardinality Features

核心想法：

对 `vocab_size > emb_skip_threshold` 的字段，不再直接置零，也不恢复完整大 embedding，而是使用固定桶数的 hash embedding。

理由：

- `emb_skip_threshold=5M` 有收益，说明被 1M 跳过的字段含有效信号。
- `7M` 参数量翻倍且 eval 下降，说明完整恢复高基数字段不可控。
- hashing trick 是推荐系统处理超高基数 ID 的常见折中：保留部分区分度，同时控制显存。

可行方案：

1. 对 skipped seq feature 使用共享或 per-field hash table，例如 `hash_bucket_size=500k/1M`。
2. hash 后仍走原 `_embed_seq_domain` concat/proj 流程。
3. 先只对 `seq_b/seq_c` 中被 skip 的字段启用，避免全局风险。

成本与风险：

- 成本中等，需要同步 train/eval model。
- 风险是 hash collision 引入噪声，但比 7M 完整 embedding 更可控。

### N2. Target-Conditioned Sequence Pooling

核心想法：

现在 query generator 用 `seq_mean`，没有显式看“当前 item 与历史行为的关系”。可以用 target item token 对序列做轻量 attention / gating，生成 target-conditioned seq summary。

理由：

- PCVR 强依赖当前 item 与用户历史兴趣是否匹配。
- P2 的 pair dense 是手工统计版本；target-conditioned pooling 是模型版。
- 参数量较小，可作为 query generator 的替换或增强，不需要加深 HyFormer。

可行方案：

1. 从 item NS tokens 聚合出 target context。
2. 对每个 seq domain 做 `attention(item_context, seq_tokens)` 得到 target-aware summary。
3. 用该 summary 替代或拼接原 `seq_mean` 生成 queries。

成本与风险：

- 成本中等，需同步 train/eval model。
- 风险低于加深层数，因为只增强 query 输入。

### N3. 无标签 Test-Aware 诊断评分卡

核心想法：

构建一个不使用 test label 的诊断脚本，把 train/valid/test 的特征分布差异量化，作为 official eval 前的候选过滤器。

理由：

- official eval 配额有限。
- valid AUC 代理失效。
- 我们已经能在 eval 日志中看到 test 时间、小时、序列长度、time bucket、zero rate 等诊断，但还没有系统化使用。

可行方案：

1. 扩展 `inspect_pcvr_structure.py`，输出 train/valid/test 的统一诊断表。
2. 指标包括：timestamp span、hour/week 分布、seq_len 分布、time_bucket 分布、zero_rate、OOV/clip/skip 覆盖率。
3. 给每个候选实验打“test-alignment 风险标签”，降低盲目 eval。

成本与风险：

- 成本低，不影响模型。
- 不能直接提分，但能节省 eval 配额并减少错误路线。

## 7. 实施路线图

### 阶段 0：立即执行

1. official eval `sparse_lr=0.08`。
2. 修正 `train/run.sh` active config，让它回到当前最佳基线。
3. 每次提交前按“平台代码更新清单”确认是否需要同步 eval。

### 阶段 1：低成本优化器线

目标：验证外部高分经验中的 optimizer / warmup 是否可迁移。

工作：

- 实现 dense lr warmup。
- 跑 `warmup_epochs=0.5/1.0` 中一个最稳配置。
- 若 `sparse_lr=0.08` official 有收益，再组合 sparse lr 与 warmup。

交付：

- 一组训练日志。
- 若校准健康，则一次 official eval。

### 阶段 2：特征工程主线

目标：补足当前模型对 test 时间窗口、recent behavior、item-history match 的表达不足。

工作顺序：

1. P1：sequence recency / interval stats。
2. P2：item-pair dense features。
3. 如果二者都有效，再考虑合并。

交付：

- train/eval 同步代码。
- 每个方向单独实验日志。
- official eval 只提交校准健康者。

### 阶段 3：结构增强与高基数字段处理

目标：在特征线稳定后，再做更强结构探索。

候选：

1. N1：hash embedding for skipped high-card features。
2. N2：target-conditioned sequence pooling。
3. P3：修正后的 LongerEncoder。

原则：

- 每次只改一个主变量。
- 不再做 `threshold/seq length/model depth` 的粗暴组合搜索。

## 8. 实验准入规则

一个模型进入 official eval 前，至少满足：

1. 与当前主基线相比，变量清晰可归因。
2. validation 没有崩：AUC、LogLoss、Brier 至少合理。
3. `prob_mean` 不明显偏离 `label_mean`。
4. 推理耗时不明显超预算。
5. 若结构改动，确认 `eval/` 已同步。

明确不建议消耗 eval 的情况：

- 只提升 validation AUC，但 LogLoss/校准明显变差。
- 与历史失败路线重复。
- 同时混入多个变量，无法判断收益来源。
- `run.sh` active config 未确认。

- `train/dataset.py`、`train/model.py`、`train/trainer.py`：数据、模型、训练 loop 的代码现实。
- `eval/infer.py`、`eval/dataset.py`、`eval/model.py`：官方推理兼容约束。

## 12. 最后原则

后续优化的主线应从“换结构、堆长度、换验证集”转向“让模型看到更贴近转化决策的信息”：

```text
optimizer / warmup
> sequence recency / interval stats
> item-pair compatibility
> high-cardinality skipped-feature hashing
> target-conditioned sequence pooling
```

每一次 official eval 都应该回答一个明确问题，而不是验证一个混合包。
